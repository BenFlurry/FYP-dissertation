\section{Application Design Overview}
\subsection{Concurrency}
To begin, the largest factor in the design of the application is the concurrency pipelining. This design separates lengthy model loading and inference times from each other and the UI thread to ensure the application remains responsive and model inference remains timely, addressing project requirements F2, NF1 and NF4. This would be impossible on a single thread as the audio processing alone would block the main UI thread. Based on the requirements, the following concurrent architecture was designed.

\begin{itemize}
    \item A \textbf{UI thread} managed by the WinRT framework, handling all rendering and user interaction.
    \item An \textbf{audio capture thread} that continuously reads from the microphone and publishes to a shared with the transcription pipeline throughout the conversation.
    \item A \textbf{transcription thread} that consumes buffered audio every 30 seconds, runs inference via Whisper-Medium then diarisation via ECAPA-TDNN, and dispatches labelled text back into a buffer in UI thread.
    \item A \textbf{summarisation thread} that loads the Med42-8B summarisation model into the GPU from program launch, then runs Med42-8B inference after the conversation ends, dispatching the result to the UI thread on completion.
    \item A \textbf{RAG thread} that loads MedCPT from program launch, then when required computes the summary embedding, queries the HNSWLib index, and retrieves matched guidance chunks.
\end{itemize}

This threading model also maps naturally onto the heterogeneous compute resources of the Intel AIPC with the transcription and diarisation models running on the CPU, the summarisation model running on the GPU, and the embeddings model running on NPU.

\subsection{Class Overview}
Figure~\ref{fig:uml_brief} illustrates the high-level object-oriented architecture of the application, demonstrating how the concurrent threads detailed in the previous section map to specific C++ classes. The architecture follows a modular approach, orchestrated by the \texttt{MainWindow} class executing on the UI thread. 

The architecture is divided into the 4 main pipelines required to handle transcription, summary, RAG and UI orchestration. 
\begin{itemize}
    \item \textbf{Audio-Transcription Pipeline (Pink, Yellow and Orange):} This pipeline handles all recording, transcription and components. The interactions within the pipeline are as follows:
    \begin{itemize}
        \item \textbf{Audio Recording and Processing (Orange)}: The \texttt{AudioRecorder} class handles all audio recording, generating an \texttt{AudioStructure} containing 30 second chunks of audio data and additional metadata required for transcription and diarisation. Every 30 seconds, the UI orchestrator thread signals for this \texttt{AudioStructure} to be published into the \texttt{AudioTranscriptionBridge}.
        \item \textbf{Audio Transcription Bridge (Yellow)}: A thread safe buffer holding the previous and next 30 seconds of audio, preventing transcription from blocking audio recording.
        \item \textbf{TranscriptionEngine (Pink)}: Subscribes to the \texttt{AudioTranscriptionBridge} to consume 30 second chunks of audio,  handling transcription then diarisation using the \texttt{DoctorEmbedding}, then appending the diarised transcript to the full transcript within the class.
    \end{itemize}
    \item \textbf{SummarisationEngine (Green)}: A class to handle all summarisation, receiving a transcript and generating a summary, and signalling to the \texttt{MainWindow} once complete. This class also sends a signal to the \texttt{MainWindow} indicating once the model has loaded in case the conversation finishes first.
    \item \textbf{RAG (Pale):} Document chunking, vector indexing, and SQLite database interactions are isolated within the \texttt{GuidelineIndexer} and \texttt{GuidelineRAG} classes, returning the \texttt{GuidelineResult} to the \texttt{MainWindow} to be displayed.
    \item \textbf{UI and Orchestration (Blue)}: Controls all threads, their lifetimes and signals for the start and end of the recording pipeline, start of summarisation and start of fetching guidelines. The \texttt{MainWindow} class is broken into several sub-classes for each UI component, which acts as a controller for their corresponding models.
\end{itemize}

\section{Environment}
The selection of Visual Studio as the primary development environment was motivated by two factors. Firstly, the integrated MSIX packaging project template facilitated streamlined deployment, including sideloading and publishing. Secondly, the IDE provided a graphical interface for configuring compiler and linker paths for multiple third-party libraries within the MSIX sandbox.

\subsection{Packaging, Publishing and Deployment}
The solution architecture utilises the MSIX packaged project template, dividing the workspace into a primary source code project, \texttt{ClinicalSummarisation}, and a dedicated packaging project, \texttt{ClinicalSummarisation (Package)}. This structural separation isolates the compilation of the application logic from the generation of the final deployment payload. 

This approach provides flexibility regarding deployment type. By "deploying" the application, it cleans, builds and loads the application onto the device to be ran through the Windows start menu. By "Publishing" the app with the self sign dialog, a \texttt{MSIXBUNDLE} file is created with a self-signed certificate to be side-loaded onto other devices. By "Publishing" the app with the Microsoft Store dialog, publisher crendentials are added to create a \texttt{MSIXBUNDLE} that can be uploaded to the Microsoft Store upload page to be published on the app store. \subsection{Library Linking}
The ML model files were housed directly within the packaging project directory and assigned a copy execution directive within the file properties. This configuration instructs the build system to embed the large model assets directly into the generated application package. Upon deployment, the models are automatically unpacked into the application's local installation directory. This design decision guarantees the models are immediately accessible to the application at runtime

Managing third-party dependencies in C++ requires configuration of header includes and library, binaries and DLL linkages. To maintain a structured codebase \texttt{External} directory at the root of the solution. The Visual Studio GUI was used to define the include directories and linker inputs for these external packages. This approach ensures that the compiler correctly resolves external API calls during the build phase and that the necessary DLLs are mapped correctly. By explicitly defining these paths within the project properties, the build system ensures that all libraries are bundled into the MSIX sandbox. This prevents runtime module execution failures when the application is deployed to an external environment, as the sandbox guarantees the environment contains identical dependencies to the development environment.
