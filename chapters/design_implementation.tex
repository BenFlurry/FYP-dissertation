\section{Application Design Overview}
\subsection{Concurrency}
To begin, the largest factor in the application design is the concurrency pipelining. This design separates lengthy model loading and inference times from other models and the UI thread. This ensures the application remains responsive, addressing project requirements F2, NF1 and NF4. This would not be possible on a single thread as the audio processing alone would block the main UI thread. Based on the requirements, the following concurrent architecture was designed.

\begin{itemize}
    \item A \textbf{UI thread} managed by the WinRT framework, handling all rendering and user interaction.
    \item An \textbf{audio capture thread} that continuously reads from the microphone and publishes to a shared buffer with the transcription pipeline throughout the conversation.
    \item A \textbf{transcription thread} that consumes buffered audio every 30 seconds, runs inference via Whisper-Medium then diarisation via ECAPA-TDNN, and dispatches a labelled transcription into the summarisation thread
    \item A \textbf{summarisation thread} that loads the Med42-8B summarisation model into the GPU from program launch, then runs Med42-8B inference upon receiving a transcript, dispatching the result to the UI thread on completion.
    \item A \textbf{RAG thread} that loads MedCPT from program launch, then awaits a summary from the UI thread. It then computes the summary embedding, queries the HNSWLib index, and retrieves matched guidance chunks.
\end{itemize}

\subsection{Class Overview}
Figure~\ref{fig:uml_brief} illustrates the high-level object-oriented architecture of the application, demonstrating how the concurrent threads detailed in the previous section map to specific C++ classes. The architecture follows a modular approach, orchestrated by the \texttt{MainWindow} class executing on the UI thread. 

The architecture is divided into the 4 main components along with their corresponding colour in the UML diagram \ref{fig:uml_brief} 
\begin{itemize}
    \item \textbf{Audio-Transcription Pipeline (Pink, Yellow and Orange):} This pipeline handles all recording, transcription and diarisation. The interactions within the pipeline are as follows:
    \begin{itemize}
        \item \textbf{Audio Recording and Processing (Orange)}: The \texttt{AudioRecorder} class handles all audio recording, generating an \texttt{AudioStructure} containing 30 second chunks of audio data and additional metadata required for transcription and diarisation. Every 30 seconds, this is published into the \texttt{AudioTranscriptionBridge}.
        \item \textbf{Audio Transcription Bridge (Yellow)}: A thread safe buffer holding the previous and next 30 seconds of audio, preventing transcription from blocking audio recording.
        \item \textbf{TranscriptionEngine (Pink)}: Subscribes to the \texttt{AudioTranscriptionBridge} to consume the audio chunks, handling transcription then diarisation using the \texttt{DoctorEmbedding} to compute the cosine similarity, then appending the diarised transcript to the full transcript held as a class member
    \end{itemize}
    \item \textbf{SummarisationEngine (Green)}: A class to handle all summarisation, receiving a transcript from the audio-transcription pipeline and generating a summary, and signalling to the \texttt{MainWindow} once complete. This class also sends a signal to the \texttt{MainWindow} indicating once the model has loaded in case the conversation finishes first.
    \item \textbf{RAG (Pale):} Document chunking, vector indexing, and SQLite database interactions are isolated within the \texttt{GuidelineIndexer} and \texttt{GuidelineRAG} classes, returning the \texttt{GuidelineResult} to the \texttt{MainWindow} to be displayed.
    \item \textbf{UI and Orchestration (Blue)}: Controls the instantiation of all pipelines and orchestrates the start of the audio-transcription-summarisation and retrieval pipelines. The \texttt{MainWindow} class is segmented into a translation per functionality
\end{itemize}

\subsection{Orchestration}
Although the UI class controls the threading and flow of this application, the sub-class architecture ensures this class is not overloaded with different purposes. This is handled through the naming convention: \texttt{MainWindow.<ComponentName>.xaml.cpp} combined with the \texttt{namespace winrt::ClinicalSummarisation::implementation} declaration, which results in 1 \texttt{MainWindow} class with multiple implementation files for ease of development. The full design and implementation can be found in section \ref{di:orchestration}

\section{Environment}
The selection of Visual Studio as the primary development environment was motivated by two factors. First, the integrated MSIX packaging project template facilitated streamlined deployment, including sideloading and publishing. Secondly, the IDE provided a graphical interface for configuring compiler and linker paths for multiple third-party libraries within the MSIX sandbox.

\subsection{Packaging, Publishing and Deployment}
The solution architecture utilises the MSIX packaged project template, dividing the workspace into a primary source code project, \texttt{ClinicalSummarisation}, and a dedicated packaging project, \texttt{ClinicalSummarisation (Package)}. This structural separation isolates the compilation of the application logic from the generation of the MSIX sandbox and linking of external libraries. This approach provides flexibility regarding deployment type. By "deploying" the application, the platform cleans, builds and side-loads the application onto the device to be ran through the Windows start menu. By "Publishing" an \texttt{MSIXBUNDLE} file is created. Certification is handled through a self-signing certificate for side loading, or a publisher's certificate for release

\subsection{Libraries}
\label{app:libraries}
\subsubsection{MiniAudio - Audio Recording}
To handle audio streaming and microphone input selection within the C++ backend, the miniaudio library was chosen. As an open-source, industry-standard tool, it reliably handles playback, capture, encoding, decoding, and data conversion. Its architecture as a single-header, dependency free C library allows for seamless integration. Implementation requires only a single macro definition to be instantiated within the project's scope, minimizing configuration overhead~\cite{mackminiaudio}.

\subsubsection{MuPDF - PDF Chunking and Viewing}
MuPDF is a lightweight and highly versatile library selected to embed a functional PDF viewer directly within the WINUI3/WinRT front-end. This is used to display relevant medical guidance documents, render bounding boxes around contextually important medical guidance sections, and support programmatic navigation between specific pages. Crucially, it achieves this while maintaining standard PDF viewer functionalities such as searching, saving, and scaling~\cite{mupdf}.

\subsubsection{SQLite3 - Embeddings Storage}
SQLite3 was implemented to store document chunk IDs, corresponding text, bounding box coordinates, PDF metadata, and the vector embeddings of user-uploaded medical guidance. By storing these embeddings alongside the raw text, SQLite3 effectively maps the semantic guidance retrieved during the vector search directly back to its source location within the raw PDF. Because this application operates entirely offline, traditional client-server relational databases such as PostgreSQL were unnecessary. Such databases would introduce unecessary network configuration and resource overhead. A serverless, self-contained SQL database keeps the entire retrieval pipeline strictly localised, ensuring zero-configuration access to indexed medical guidance~\cite{sqlite}.

\subsubsection{HNSWLib - Retrieval Search}
To power the RAG pipeline, an Approximate Nearest Neighbor (ANN) approach was required. While exact match retrieval algorithms such as kNN offer high precision, their linear scaling results in linearly slow inference times for large document sets. HNSWLib was selected as it prioritises retrieval speed without significantly sacrificing the semantic accuracy which is vital for medical contexts. It was chosen over heavier, managed vector databases to ensure the application's local footprint remained lightweight and responsive~\cite{malkov2018efficient}.

\subsection{Models}
\subsubsection{Med42-8B - Summarisation}
For the core summarization task, the Llama3 based LLM Med42-8B model pre-trained on biomedical textbooks and PubMed publications was selected. This model was chosen following a comparative analysis of the leading sub-10b parameter medical LLMs \ref{sec:med42test}. Utilising a domain-specific pre-trained LLM increases the accuracy of responses within this domain\cite{nissen2025medicine}. While the 70B model used in predecessor to this project was considered, hardware limitations of the target Intel AIPCs (32GB system RAM and 18GB VRAM) prevented loading and quantising such a large model locally. The 8B variant serves as the optimal intersection of hardware compatibility and summarisation accuracy~\cite{med42_2023}.

\subsubsection{Whisper-Medium - Transcription}
Speech-to-text transcription is handled by Whisper-Medium. This open-source model was trained on a highly diverse, multi-lingual dataset, cleanly enabling future multi-lingual support. Furthermore, OpenVINO only natively supports Whisper architectures within its \texttt{WhisperPipeline} class, this was the only viable model family. Testing across the Whisper family revealed that the "Large" variant introduced too much latency and performance overheads when running on the CPU, whereas the "Medium" model provided the ideal balance of inference speed and transcription accuracy compared to the "Base", "Normal" and "Tiny" models. Its efficiency allows the application to transcribe conversations in real time, eliminating the need to buffer and store audio files for post-processing~\cite{radford2023robust}.

\subsubsection{Speechbrain ECAPA-TDNN - Diarisation}
To generate labelled transcripts, a model is required to handle speaker diarisation, the process of identifying which participant is speaking at any given time. This is achieved by passing sentences of the raw audio and transcript into the ECAPA-TDNN model, which calculates the voice embeddings for each sentence. The system mathematically compares the embeddings of an unknown speaker against a pre-recorded embedding of the doctor's voice to determine whether the doctor or patient is speaking for a given sentence. This allows the application to definitively label each transcribed sentence as \texttt{[Doctor]} or \texttt{[Patient]}. Because the model is exceptionally lightweight, it runs efficiently on the CPU concurrently with the Whisper model and the rest of the application during conversation, significantly reducing post-processing delays before summarisation begins~\cite{desplanques2020ecapa}.

\subsubsection{MedCPT - Guidance Embeddings}
MedCPT is a domain-specific semantic information retrieval model pre-trained exclusively on biomedical data. In this pipeline, it is utilised to convert both the generated conversation summaries and the raw PDF guidelines into dense vector representations. These vectors are then mathematically searched with HNSWLib to retrieve the closest contextual match between the consultation summary and the uploaded medical guidance. By harnessing dense semantic matching rather than simple lexical matching, the system ensures that the clinical meaning behind a patient's symptoms or a doctor's recommendation is linked to the correct guidance~\cite{jin2023medcpt}.


\subsection{Library Linking}
The ML model files were housed directly within the packaging project and assigned a copy execution directive within the file properties. This configuration instructs the build system to embed the large model assets directly into the generated application package. Upon deployment, the models are unpacked into the MSIX sandbox.

Managing third-party dependencies in C++ requires configuration of header includes and library, binaries and DLL linkages. To maintain a structured codebase \texttt{External} directory at the root of the solution. The Visual Studio GUI was used to define the include directories and linker inputs for these external packages. This approach ensures that the compiler correctly resolves external API calls during the build phase and that the necessary DLLs are mapped correctly. By explicitly defining these paths within the project properties, the build system ensures that all libraries are bundled into the MSIX sandbox. This prevents runtime module execution failures when the application is deployed to an external environment, as the sandbox guarantees the environment contains identical dependencies to the development environment.
